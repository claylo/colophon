# Handoff: Phase 2 Curate Pipeline

**Status:** Yellow — code complete, 144 tests passing, but `claude -p` invocation is slow. Needs debugging — likely an issue with how the invocation is constructed, NOT with the CLI itself.
**Date:** 2026-03-10
**Branch:** `main`

## Where things stand

Phase 2 (Curate) infrastructure is fully built and tested: types, post-processing (location mapping, hierarchy inversion, main-file flagging), CLI wiring with `-- <claude-args>` passthrough, JSON Schema, compact prompt format. The `claude -p --json-schema` invocation works but takes 7+ minutes for 500 candidates — unacceptable. The bottleneck is the Claude Code agent runtime overhead, not the model itself.

Phase 1 improvements also landed this session: tracing instrumentation at all pipeline decision points, heading exclusion from prose, `max_doc_pct` filter (default 0.7), 10 new tests for known_terms and regex exclude, updated config examples. Test count went 111 → 144.

## Decisions made

- **Headings excluded from prose extraction** — they inject short, generic noise into YAKE. Body prose captures anything indexable.
- **`max_doc_pct` = 0.7 default** — terms in >70% of docs are definitional (what the book is *about*), not indexable. Filter runs after known_terms consolidation.
- **Compact prompt format** — `term | score | file1, file2` instead of full YAML with context snippets. Reduces 642KB → ~25KB. In `curate/claude.rs::build_compact_candidates()`.
- **Schema as YAML source** — `config/curate-schema.yaml` authored in YAML, converted to JSON via `yj`, embedded at compile time via `include_str!`.
- **`-- <claude-args>` passthrough** — clap `last = true` forwards arbitrary args to the claude CLI invocation.
- **`fastMode` via `--settings`** — `--settings '{"fastMode": true}'` enables fast mode for `claude -p`.

## What's next

Priority: **debug and fix the invocation speed.** The `claude -p` CLI is a thin API passthrough — the problem is in how we're calling it, not the tool itself.

1. **Test minimal invocation** — `unset CLAUDECODE && claude -p "Return one term" --json-schema '{...tiny...}' --model sonnet --output-format json --max-turns 1` with a trivial schema. Verify it completes in seconds.

2. **Isolate the slow part** — schema description bloat? Inline `--system-prompt` with large string? Try `--system-prompt-file` or `--append-system-prompt` instead. Strip JSON Schema `description` fields to bare minimum.

3. **Scale up** — once minimal case works fast, gradually add candidates and measure.

4. **Do NOT pivot to ureq** until the CLI approach has been properly debugged. Read the CLI docs (`~/source/claude/docs/`) before making assumptions.

## Landmines

- **`CLAUDECODE` env var** — must be unset when spawning `claude` from within a Claude Code session, or it refuses to start. Already handled in code with `cmd.env_remove("CLAUDECODE")`.
- **`-c` short flag clash** — `--candidates` on the curate subcommand can't use `-c` because the global `--config` owns it. Already fixed (uses `--candidates` long form only).
- **`config/curate-schema.json` is a build artifact** — regenerate with `yj < config/curate-schema.yaml > config/curate-schema.json` after editing the YAML source. Not automated via build.rs yet.
- **`serde_json` added to colophon-core** — needed for parsing Claude CLI JSON response. Added via `cargo add`.
- **`which` crate added** — for finding `claude` in PATH. Version 7, MIT licensed, deny-clean.
