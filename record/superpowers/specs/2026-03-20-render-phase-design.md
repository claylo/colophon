# Phase 3: Render

**Date:** 2026-03-20
**Status:** Approved

## Summary

`colophon render` reads the curated term database from Phase 2 and produces
two outputs: annotated copies of source files with inline index markers, and
an optional standalone glossary document. A `Renderer` trait abstracts the
output format; the first (and currently only) implementation targets Typst
with in-dexter index markers.

## Data Flow

```
colophon-terms.yaml
        │
        ▼
  ┌─────────────┐
  │ render::run │  read terms, read source files, match locations
  └──────┬──────┘
         │
         ├── per source file: Renderer::annotate()
         │       insert #index markers at term locations
         │       write annotated copy to output dir
         │
         └── if --glossary: Renderer::glossary()
                 emit standalone glossary document
```

## Trait

```rust
/// A single index marker to place in a source file.
pub struct Annotation {
    /// Canonical term text.
    pub term: String,
    /// Parent chain from root to immediate parent (empty if top-level).
    /// Example: for "OAuth" with parent "authentication" and grandparent
    /// "security", this would be ["security", "authentication"].
    pub parent_chain: Vec<String>,
    /// Whether this is a substantive discussion (bold page number).
    pub main: bool,
    /// Byte offset in the source where this marker should be inserted
    /// (immediately after the term occurrence).
    pub byte_offset: usize,
}

/// Format-specific rendering of index markers and glossary.
pub trait Renderer {
    /// Insert index markers into source file content.
    ///
    /// `annotations` are pre-sorted by descending position so insertions
    /// don't shift subsequent offsets. Returns the annotated content.
    fn annotate(&self, source: &str, annotations: &[Annotation]) -> String;

    /// Emit a standalone glossary document from the curated terms.
    fn glossary(&self, terms: &CuratedTermsFile) -> String;
}
```

## Annotation Placement

The orchestrator (`render::run`) handles location matching:

1. For each source file, collect all terms with a matching `location.file`.
2. For each term-location pair, find the first occurrence of the term text
   in the source content using case-insensitive byte-offset search (a new
   function; `extract::markdown::extract_context` returns context snippets
   but not offsets). Also search for each alias in `term.aliases` — alias
   matches generate an annotation using the canonical term name.
3. Record the byte offset immediately after the matched term text.
4. Build an `Annotation` with the pre-computed parent chain (walk parent
   pointers, detect cycles, reverse into root-first order).
5. Sort annotations by descending byte offset.
6. Pass to `Renderer::annotate()`, which inserts markers without
   offset-shifting issues.

One marker per term per file. In-dexter collects page numbers, so
multiple markers for the same term in one file would produce duplicate
page entries. A future iteration could support section-level granularity
if source files span multiple chapters.

Terms that cannot be located in the source file are logged as warnings
and skipped. This handles suggested terms (no locations) and stale
context gracefully.

## Typst Renderer

### Index Markers

| Scenario | Output |
|----------|--------|
| Top-level term, regular mention | `#index[OAuth]` |
| Top-level term, main discussion | `#index-main[OAuth]` |
| Child term, regular mention | `#index("authentication", "OAuth")` |
| Child term, main discussion | `#index-main("authentication", "OAuth")` |
| Grandchild term | `#index("security", "authentication", "OAuth")` |

Markers are inserted immediately after the first occurrence of the term
in each source file. `#index-main` is a convenience export from in-dexter
defined as `index.with(fmt: strong)` — it inherits all parameters
including positional nesting args. The `#index` call is invisible in
rendered output (it only records metadata for the index page).

### Import Preamble

Annotated files that receive at least one marker get an in-dexter import
prepended (if not already present):

```typ
#import "@preview/in-dexter:0.7.2": *
```

### Glossary

Emitted as a native Typst term list:

```typ
// Generated by colophon

/ API key: A credential used to authenticate requests to an API.
/ authentication: The process of verifying user or system identity.
  / OAuth: An open standard for token-based authorization.
    _See also: API key_
/ TLS: Transport Layer Security protocol for encrypted connections.
```

- Top-level terms at root indentation.
- Children indented under their parent.
- `see_also` rendered as italic "See also: X, Y" after the definition.
- Sorted alphabetically, hierarchy preserved.

## CLI

```
colophon render [OPTIONS]

Options:
    --terms <FILE>       Path to curated terms file [default: colophon-terms.yaml]
    --format <FORMAT>    Output format [default: typst] (only typst for now)
    --output-dir <DIR>   Output directory [default: .]
    --glossary           Also emit a standalone glossary document
    --json               Output as JSON (file list + stats)
```

## Module Layout

```
crates/colophon-core/src/render/
├── mod.rs          # Renderer trait, Annotation type, run() orchestrator
└── typst.rs        # TypstRenderer: annotate() + glossary()

crates/colophon/src/commands/render.rs  # CLI args + wiring
```

Adding a new format means one new file implementing `Renderer` and a
match arm in `run()`. No trait objects needed — the format is known at
CLI parse time, so static dispatch via an enum works.

## Config

No new config section for now. The render command uses:
- `source.dir` and `source.extensions` from existing config (to find source files)
- CLI flags for everything else

A `[render]` config section can be added later if defaults need persistence.

## Testing Strategy

- **Unit tests (typst.rs):** Known annotations + source text -> expected
  annotated output. Verify marker format for all scenarios (top-level,
  child, grandchild, main/regular).
- **Glossary tests:** Known CuratedTermsFile -> expected Typst term list.
  Verify hierarchy indentation, see_also rendering, alphabetical sort.
- **Parent chain tests (mod.rs):** Walk parent pointers, verify chain
  construction. Test cycle detection (A -> B -> A) and dangling parents.
- **Integration test:** Small curated terms file + source .typ files ->
  verify annotated output contains expected markers and glossary is valid.
- **3-level hierarchy test:** As specified in
  [ADR 0001](../../record/decisions/0001-index-hierarchy-depth-and-render-mapping.md),
  verify grandparent -> parent -> child reconstructs into
  `#index("gp", "p", "c")`.

## Error Handling

`render::run` returns `Result<RenderOutput, RenderError>` where `RenderError` covers:
- Terms file not found or malformed YAML
- Source directory missing or unreadable
- Output directory not writable
- Cycle detected in parent chains (A -> B -> A)

Per-file warnings (term not found in source, alias not found, duplicate
term in file) are logged via tracing and included in the summary stats,
not treated as fatal errors.

## Notes

- The in-dexter import version (`0.7.2`) is hardcoded in the initial
  implementation. This should become configurable (in `[render]` config
  or CLI flag) once the render config section is added.

## Not In Scope

- LaTeX, Markdown, or JSON renderers (Phase 4)
- `colophon generate` all-in-one command (Phase 4)
- In-place source modification (always writes copies)
- `#make-index()` page generation (user controls this in their document)
- `colophon init` scaffolding
